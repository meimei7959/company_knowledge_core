# Phase 2 方案二：部署中枢与多设备 Runner 工作台产品补充

## 一句话目标

把项目中枢部署成一个“登记入口可操作，执行监管只读”的工作台，让 Owner 能创建项目、邀请或注册电脑、登记或申请工具，并实时看见接入了几台电脑、覆盖几个项目、每个项目的任务进展、每台电脑正在用哪个 Agent / 工具 / Codex / Claude 执行什么，以及这些状态是否来自真实双机 Runner 上报。

## 背景

既有 Phase 2 多设备 Runner 文档已经定义了多设备协作闭环：Runner 注册、任务租约、上下文拉取、执行写回、审计和通知。本补充补齐方案二的产品可见性、登记入口和验收口径：中枢先作为可部署、可观察、可登记的项目运行工作台，但不在工作台里直接派单、执行、修复、转派、篡改任务结果或绕过验收。

方案二的核心不是证明“一个本地进程能跑”，而是证明两台真实电脑接入同一个中枢后，中枢能可靠显示多项目、多任务、多 Runner 的当前事实，并且不会越权展示或混写项目数据。

## 产品范围

### 本期必须有

1. 工作台总览：显示当前可见电脑数量、项目数量、任务数量、风险数量和最近更新时间。
2. 项目列表与项目详情：显示每个项目的需求、任务、进展、Runner、Agent、模型和 token 使用。
3. 电脑列表与电脑详情：显示每台电脑的注册状态、心跳、可执行 Agent、可用工具、Codex / Claude / 本地模型能力、当前任务和历史任务摘要。
4. 创建项目入口：Owner 能在工作台登记新项目、项目 Owner、初始需求、资料入口和治理规则，形成可审计项目记录。
5. 电脑注册 / 邀请流程：Owner 能邀请同事电脑接入；Runner 管理员能提交注册申请；授权后才能进入可调度状态。
6. 工具注册 / 申请流程：工具 Owner 或 Runner 管理员能登记工具、申请工具可用范围，审批后才能用于任务。
7. Runner 进展上报：每台电脑能向中枢上报心跳、租约、任务状态、执行进展、模型使用、token 使用、工具调用摘要、结果和错误。
8. 项目隔离与权限边界：不同项目、不同角色、不同 Runner 只能看到或登记授权范围内的信息；登记类写操作必须审批、确认并审计。
9. 真实双机验收：至少两台不同电脑同时接入同一中枢，分别领取或执行不同任务，中枢能准确展示状态、隔离和异常恢复。

### 本期明确不做

1. 不在工作台中提供远程控制电脑、远程 shell、远程点击、远程打开 Codex / Claude 的能力。
2. 不在工作台中直接派单、领取任务、修复失败任务、续租、转派、暂停执行中 Runner、重跑任务、改写任务状态或提交验收。
3. 不实现 Agent Ring 本地运行时本身，只定义中枢需要看到和验收的注册、上报、隔离和只读展示口径。
4. 不把 token 统计作为计费系统；本期只用于项目观测、预算预警和执行复盘。
5. 不允许任何用户在工作台直接篡改 TaskResult、AgentRun、证据引用、测试结果、token 记录或审计记录。

## 用户与使用场景

| 用户 | 要看的问题 | 产品响应 |
| --- | --- | --- |
| 项目 Owner | 现在有几台电脑在帮我做几个项目？每个项目卡在哪里？ | 总览显示电脑数、项目数、任务状态分布、阻塞项和最近上报时间。 |
| 项目经理 Agent | 哪个需求下面有哪些任务？由谁执行？用了什么模型？token 是否超预算？ | 项目详情按需求分组展示任务明细、Runner、Agent、模型、token 和下一步。 |
| Runner 管理员 | 我的电脑是否注册成功？正在做什么？中枢看到什么能力？ | 电脑详情显示注册状态、授权项目、能力、工具、当前 Agent 和心跳。 |
| 工具 Owner | 某个工具能否登记给项目或 Runner 使用？谁批准？范围是什么？ | 工具注册流程记录工具能力、风险、Owner、适用项目、审批状态和审计。 |
| 系统管理员 | 哪台电脑越权、离线或上报异常？ | 电脑列表显示授权范围、心跳、异常、版本、审计入口和只读诊断信息。 |
| 测试 Agent | 双机是否真的成立？隔离是否可证明？ | 验收面板或验收记录显示两个不同机器指纹、并发任务、隔离失败用例和证据。 |

## 工作台操作边界

1. 工作台允许登记类操作：创建项目、邀请电脑、提交电脑注册、登记工具、申请工具可用范围、提交审批确认。
2. 工作台执行监管只读：任务匹配、任务领取、执行进展、任务结果、AgentRun、证据、测试结果、token、审计只能展示，不能在工作台直接修改。
3. 登记类操作必须写入 `AuditLog`，并关联操作者、时间、对象、前后状态、审批人、确认记录和影响范围。
4. 需要审批的登记类操作在审批通过前只能进入 pending 状态，不得自动产生可调度 Runner、可用工具或 active 项目权限。
5. 工作台不得直接触发任务领取、任务完成、执行修复、Runner 执行暂停、租约续期、任务转派、仓库写入、知识发布或验收通过。
6. 可见不等于全量可见。用户只能读或登记自己有权限的项目、任务、Runner、工具和证据摘要。
7. 任何登记入口都必须提供确认页，确认页用人能读懂的名称、范围、风险和审批路径，不用内部 ID 作为主说明。

## 登记入口用户流程

### 创建项目流程

| 步骤 | 用户动作 | 系统行为 | 权限 / 审批 / 审计 |
| --- | --- | --- | --- |
| 1 | Project Owner 在工作台点击创建项目。 | 展示项目名称、Owner、目标、初始需求、资料来源、可见范围、默认治理规则表单。 | 只有具备项目创建权限的用户可进入；无权限用户只看到只读项目列表。 |
| 2 | Owner 填写初始需求和资料入口。 | 创建 `Project` 草稿和初始需求记录；不自动创建可执行任务。 | 写 `AuditLog`：created_project_draft，记录操作者、字段摘要和资料来源。 |
| 3 | Owner 确认项目范围。 | 展示确认页：项目可见成员、可用 Runner 范围、默认工具范围、知识敏感级别。 | 高敏项目、跨团队项目或包含客户资料时进入审批；普通项目可由 Owner 确认生效。 |
| 4 | 审批人批准或拒绝。 | 批准后项目进入 active 或 ready_for_planning；拒绝后保留草稿和原因。 | 写 `AuditLog` 和 `NotificationRecord`；审批记录必须可在项目详情中只读查看。 |

创建项目验收：

1. 工作台能创建项目登记记录，但不能直接生成执行租约。
2. 项目从 draft 到 active 的每次状态变化都有操作者、确认人或审批人、时间和原因。
3. 未获项目权限的用户不能看到项目详情、任务数量、Runner 明细或 token 汇总。
4. 项目创建后如需拆任务，必须进入调度或项目经理 Agent 流程，不由工作台直接派单。
5. 创建请求必须带幂等键，重复提交不得创建重复项目。

### 电脑注册 / 邀请流程

| 步骤 | 用户动作 | 系统行为 | 权限 / 审批 / 审计 |
| --- | --- | --- | --- |
| 1 | Owner 或系统管理员在工作台发起电脑邀请。 | 生成邀请记录，包含项目范围、期望 Runner Owner、有效期、允许能力和数据范围。 | 只有项目 Owner 或管理员可邀请；邀请写 `AuditLog`，邀请链接或令牌不在列表明文展示。 |
| 2 | Runner 管理员在自己的电脑提交注册申请。 | 上报 runnerId、机器指纹摘要、Owner、客户端版本、Codex / Claude / local model 能力、工具和仓库范围。 | 注册进入 `pending_authorization`；申请人只能看到自己的申请状态。 |
| 3 | Owner 或管理员审核注册申请。 | 工作台展示差异：申请范围、邀请范围、风险项、重复设备提示、健康检查状态。 | 超出邀请范围必须二次确认或拒绝；审批写 `AuditLog` 和 `NotificationRecord`。 |
| 4 | 审批通过后 Runner 进入可见状态。 | Runner 可进入 `online_readonly` 或 `online_schedulable`；是否可调度由权限和健康检查决定。 | 授权范围、有效期、审批人、能力验证状态必须可追溯。 |

电脑注册 / 邀请验收：

1. 未被邀请的电脑可以提交申请，但不会自动获得项目权限。
2. 邀请范围、申请范围、批准范围三者必须能在工作台对比。
3. 电脑注册是工作台可操作事项；任务领取和执行仍由 Agent Ring / Scheduler 处理。
4. 重复机器、过期邀请、越权申请必须进入可读错误和审计，不得静默失败。

### 工具注册 / 申请流程

| 步骤 | 用户动作 | 系统行为 | 权限 / 审批 / 审计 |
| --- | --- | --- | --- |
| 1 | 工具 Owner 或 Runner 管理员登记工具。 | 填写工具名称、用途、版本、可执行动作、风险等级、需要的项目 / Runner 范围、输入输出数据类型。 | 只有具备工具登记权限的用户可提交；登记写 `AuditLog`。 |
| 2 | 系统生成工具申请。 | 工具进入 `pending_review`，不可被任务使用；工作台显示审批人、风险、影响项目。 | 高风险工具、写权限工具、外部 API 工具必须审批；低风险只读工具可按策略快速确认。 |
| 3 | Tool Owner / 管理员审批。 | 批准后写入 `ToolAsset` 或工具授权记录；拒绝后保留原因。 | 审批记录必须包含批准范围、有效期、风险说明和撤销路径。 |
| 4 | Runner 上报工具健康。 | 工具在具体电脑上显示 healthy、degraded、failed 或 unverified。 | 工具可登记不代表每台电脑可用；电脑侧可用性必须由 Runner 上报。 |

工具注册 / 申请验收：

1. 未批准工具不得出现在任务可用工具范围内。
2. 工具批准范围必须精确到项目、Runner、Agent 角色、数据范围和有效期。
3. 工作台不能直接调用工具，只能登记、申请、审批和展示工具状态。
4. 工具授权变更必须写审计，并在受影响项目或 Runner 详情可见。

## 总览页需求

| ID | 需求 | 展示字段 | 验收口径 |
| --- | --- | --- | --- |
| P2-OBS-001 | 显示可见电脑数量。 | 总电脑数、在线数、离线数、异常数、待授权数。 | 两台电脑接入后，Owner 能看到 `2 台电脑`，并能区分在线和离线。 |
| P2-OBS-002 | 显示可见项目数量。 | 总项目数、活跃项目数、阻塞项目数、待验收项目数。 | 用户无权限的项目不计入数量，也不出现在列表。 |
| P2-OBS-003 | 显示任务状态分布。 | waiting_runner、claimed、processing、blocked、waiting_acceptance、done、failed、stale_lease。 | 状态来自中枢事实记录，不用前端自行猜测。 |
| P2-OBS-004 | 显示最近上报健康度。 | 最近心跳时间、最近进展上报时间、最近结果写回时间、最近错误。 | 任一 Runner 超过心跳阈值后，总览出现离线或异常提示。 |
| P2-OBS-005 | 显示模型与 token 总量。 | 按项目、Runner、Agent 汇总 input / output / cached / total token；展示预算阈值。 | token 缺失时显示 unknown，不用 0 掩盖；双机验收任务必须有非 unknown 统计。 |

## 项目列表与项目详情

### 项目列表字段

每个项目至少展示：

1. 项目名称、项目 Owner、项目状态。
2. 关联需求数、任务总数、进行中任务数、阻塞任务数、待验收任务数、完成任务数。
3. 已接入 Runner 数、当前执行中 Runner 数、离线 Runner 数。
4. 最近任务进展、最近 Runner 心跳、最近结果写回时间。
5. 本项目 token 使用总量、今日 token 使用量、预算预警状态。

### 项目详情：需求到任务明细

项目详情必须按“需求 -> 任务 -> 执行记录”组织，而不是只按 Runner 平铺。

| 层级 | 必须展示 | 验收口径 |
| --- | --- | --- |
| 需求 | 需求标题、需求编号或可读标识、业务状态、验收口径、关联任务数。 | Owner 能回答“这个任务属于哪个需求”。 |
| 任务 | 任务标题、任务类型、状态、优先级、所需能力、所属需求、验收策略、当前 Runner、当前 Agent、租约状态。 | 任何执行中任务都能看出谁在做、做到哪一步、下一步是什么。 |
| 执行进展 | 百分比或阶段、当前步骤、最近进展摘要、阻塞原因、下一步、最近更新时间。 | 进展必须来自 Runner 上报或 TaskResult，不允许前端生成虚假进度。 |
| 模型 | 执行引擎类型、模型名称、供应方、本地或托管、Codex / Claude / local model 标识、版本或客户端版本。 | Owner 能看出任务是 Codex、Claude 还是其他模型执行。 |
| token | input token、output token、cached token、reasoning token、total token、预算、超额状态、上报来源。 | token 按单次 AgentRun 和任务累计展示；缺失必须标记 unknown。 |
| 证据 | TaskResult、AgentRun、输出引用、检查结果、审计记录、通知记录。 | 不登录 Runner 电脑，也能从中枢看到可复盘证据。 |

### 项目任务状态解释

| 状态 | 人能读懂的含义 | 必须有的下一步 |
| --- | --- | --- |
| waiting_runner | 正在等合适电脑领取。 | 显示缺少的能力、权限或在线 Runner。 |
| claimed | 某台电脑已领取但尚未开始执行。 | 显示租约 Owner、租约过期时间。 |
| processing | Runner 正在执行。 | 显示当前步骤、Agent、工具、模型、最近上报。 |
| blocked | 任务阻塞。 | 显示阻塞原因、责任人、恢复路径。 |
| waiting_acceptance | 已写回结果，等待验收。 | 显示验收人、验收标准和结果引用。 |
| done | 任务完成。 | 显示结果、证据、检查和完成时间。 |
| failed | 执行失败。 | 显示失败原因、可重试性和责任人。 |
| stale_lease | 租约过期或 Runner 失联。 | 显示最后心跳、可恢复动作和审计记录。 |

## 电脑列表与电脑详情

### 电脑列表字段

每台电脑至少展示：

1. 电脑名称、Runner 名称、设备 Owner、注册状态。
2. 在线状态、最后心跳时间、当前负载、当前任务数。
3. 授权项目数、可见项目列表摘要、数据范围摘要。
4. 可执行 Agent 角色：如产品经理 Agent、工程 Agent、测试 Agent、知识整理 Agent。
5. 可用工具：如 git、browser、lark-cli、test runner、registered tool assets。
6. 可用执行器：Codex CLI、Claude CLI、本地模型、浏览器自动化、脚本运行器。
7. 当前工作中的 Agent、任务、项目、需求、模型和 token 累计。
8. 最近错误、版本、注册时间、最近授权变更时间。

### 电脑详情字段

| 区块 | 必须展示 | 验收口径 |
| --- | --- | --- |
| 身份 | runnerId、机器可读名称、设备 Owner、机器指纹摘要、注册来源、注册时间。 | 两台真实电脑必须有不同机器指纹摘要；不得只靠进程名区分。 |
| 授权 | 可访问项目、仓库范围、数据范围、工具范围、Agent 角色范围、有效期。 | 未授权项目不展示任务内容，只展示无权限或不计入。 |
| 运行能力 | Codex / Claude / local model 是否可用、客户端版本、模型列表、工具列表。 | 能看出这台电脑能不能跑 Codex、Claude，能用哪些工具。 |
| 当前工作 | 工作中的 Agent、任务、所属项目、所属需求、状态、当前步骤、租约过期时间。 | Owner 能回答“这台电脑现在在干什么”。 |
| 进展与资源 | 最近进展、token、运行时长、重试次数、错误摘要。 | 运行中任务每次上报后详情页更新。 |
| 历史摘要 | 最近 N 个任务、完成/失败/阻塞数量、平均完成时间。 | 只显示授权范围内历史，不泄漏其他项目。 |

## Runner 注册流程

### 注册步骤

1. Runner 管理员在自己的电脑启动 Agent Ring 或 Runner 注册流程。
2. 本机生成机器指纹摘要、runnerId 候选、设备名称、设备 Owner、客户端版本和能力声明。
3. 本机向中枢提交注册申请，包含可执行 Agent、工具、模型、仓库、数据范围和期望接入项目。
4. 中枢创建 `AgentRunner` 注册记录，初始状态为 `pending_authorization`。
5. 项目 Owner 或管理员在授权流程中确认项目范围、仓库范围、工具范围、数据范围和有效期。
6. 授权完成后 Runner 状态变为 `online_schedulable` 或 `online_readonly`。
7. Runner 开始心跳，上报可调度状态、当前负载和能力健康检查结果。

### 注册申请字段

| 字段 | 说明 | 必填 |
| --- | --- | --- |
| runnerName | 人能读懂的电脑名称。 | 是 |
| runnerId | 中枢侧唯一 Runner 标识。 | 是 |
| deviceOwner | 设备责任人。 | 是 |
| machineFingerprintHash | 机器指纹摘要，不存明文敏感硬件信息。 | 是 |
| clientVersion | Runner 或 Agent Ring 客户端版本。 | 是 |
| capabilities | codex_cli、claude_cli、local_model、browser、git、test_runner 等能力。 | 是 |
| availableAgents | 本机可执行 Agent 角色。 | 是 |
| availableTools | 本机可用工具资产和版本。 | 是 |
| availableModels | 可用模型、供应方、本地/托管标识。 | 是 |
| requestedProjects | 申请接入的项目。 | 是 |
| repositoryScopes | 申请访问的仓库范围。 | 按项目需要 |
| dataScopes | 申请访问的数据范围。 | 是 |

### 注册验收

1. 未授权 Runner 只能出现在待授权列表，不得领取任务。
2. 授权前后必须有审计记录，说明谁批准了哪些项目、工具、仓库和数据范围。
3. 注册声明不等于能力已验证；中枢必须显示健康检查状态：unverified、healthy、degraded、failed。
4. 同一机器重复注册时，中枢必须提示可能重复设备，避免把同一电脑伪装成双机验收。

## Runner 上报到中枢

### 上报类型

| 上报 | 触发时机 | 必须内容 | 中枢展示 |
| --- | --- | --- | --- |
| heartbeat | 固定间隔和状态变化时。 | runnerId、时间、在线状态、负载、当前任务、版本、最近错误。 | 电脑在线状态、最后心跳、负载。 |
| lease_update | 领取、续租、释放任务时。 | taskId、leaseToken、leaseOwner、leaseExpiresAt、状态。 | 项目任务租约状态。 |
| progress_update | 任务步骤变化、工具调用完成、模型调用完成、阻塞时。 | 当前步骤、进展摘要、Agent、工具、模型、token、阻塞原因、下一步。 | 项目详情和电脑详情实时进展。 |
| tool_usage | 使用已批准工具后。 | 工具名、版本、调用目的、结果摘要、错误、证据引用。 | 当前工具、执行证据、审计摘要。 |
| model_usage | Codex / Claude / local model 调用后。 | 执行器、模型、供应方、input/output/cached/reasoning/total token、上下文预算。 | 模型与 token 明细。 |
| result_writeback | 任务完成或终止时。 | TaskResult、AgentRun、outputRefs、evidenceRefs、testsOrChecks、qualityEvaluation、terminalReason。 | 待验收、完成、失败或阻塞状态。 |
| error_report | 任意不可恢复或需人工处理错误。 | 错误类型、可读原因、影响范围、责任人建议、恢复建议。 | 风险、阻塞、最近错误。 |

### 上报验收

1. 上报必须带 runnerId、projectId、taskId 或明确适用范围；无法归属的上报进入隔离错误队列，不更新项目事实。
2. 进展上报不得覆盖其他 Runner 的租约任务。
3. token 上报必须能关联到 AgentRun；无法关联时只进入诊断，不进入项目 token 汇总。
4. Runner 离线后，中枢必须保持最后一次可信状态，并标记数据已过期。
5. result_writeback 缺少 TaskResult 或 AgentRun 必填字段时，任务不得进入 done。

## 项目隔离

项目隔离必须同时作用于注册、任务匹配、上下文拉取、进展上报、结果写回和只读展示。

| 环节 | 隔离要求 | 验收口径 |
| --- | --- | --- |
| 注册 | Runner 只能申请项目范围，不能自授权。 | 未经 Owner 或管理员授权不进入可调度。 |
| 匹配 | 调度器只在 Runner 授权项目内匹配任务。 | Runner A 无项目 B 权限时，不能领取项目 B 任务。 |
| 上下文 | Runner 拉取的 context pack 只包含授权项目、需求、资料和验收规则。 | 项目 B 的需求、资料、路径和证据不会出现在项目 A Runner 的上下文包中。 |
| 上报 | 上报必须匹配有效租约、项目和 Runner。 | 伪造 projectId 或 taskId 的上报被拒绝并审计。 |
| 工作台 | 用户只能看到或登记自己有权限的项目、Runner 和工具范围。 | 项目数量、任务数量、Runner 数量、工具数量和 token 汇总都按权限过滤。 |
| 结果 | TaskResult 只能写回所属任务和项目。 | 跨项目写回失败，且不会污染另一个项目时间线。 |

## 权限与只读边界

### 角色权限

| 角色 | 可见 / 可操作范围 | 不允许 |
| --- | --- | --- |
| Project Owner | 查看自己项目内的电脑、任务、需求、结果、风险和指标；创建项目；邀请电脑；申请项目工具范围；确认普通项目登记。 | 查看未授权项目明细；在工作台直接派单、修复任务、改任务结果或扩大工具权限。 |
| Runner Admin | 查看自己设备的注册状态、授权范围、当前任务和历史摘要；提交电脑注册申请；提交本机工具可用性。 | 查看其他设备的敏感明细；自授权项目、仓库、数据或工具范围；直接领取任务。 |
| Tool Owner | 登记工具、维护工具说明、审批或拒绝自己负责工具的项目 / Runner 使用申请。 | 绕过项目 Owner 授权；直接调用工具执行项目任务；修改历史工具使用记录。 |
| System Admin | 查看全局 Runner 注册、授权、健康和审计摘要；审批跨项目、高敏或系统级登记申请。 | 绕过审批读取项目敏感资料原文，除非另有授权；篡改结果、证据或审计。 |
| Viewer | 查看被授权项目的只读状态。 | 创建项目、邀请电脑、注册工具、审批、执行、派单、修改任何状态。 |

### 只读展示脱敏

1. secret、token、key、cookie、私有凭据永不展示。
2. 本机路径、内部任务 ID、租约 token 不作为主要人读信息展示；必要时只在诊断区显示摘要或后四位。
3. token 统计展示数字和来源，不展示完整 prompt 或模型原始上下文。
4. 错误信息必须转成可读摘要；原始堆栈只作为受控证据引用。

## 真实双机验收标准

### 环境要求

1. 至少两台不同电脑接入同一个部署中枢，例如电脑 A 和电脑 B。
2. 两台电脑必须有不同机器指纹摘要、不同 runnerId、独立心跳和独立租约。
3. 同一台电脑启动两个 Runner 进程不算真实双机验收。
4. 两台电脑至少各自声明一种可验证执行器：Codex CLI、Claude CLI 或本地模型；其中至少一台必须上报 Codex 或 Claude 执行明细。
5. 中枢必须部署在两台电脑都能访问的位置，而不是只在某一台本机进程里读取本地文件。

### 验收数据准备

1. 创建至少两个项目，项目 A 和项目 B。
2. 项目 A 至少有一个需求，需求下至少两个可执行任务。
3. 项目 B 至少有一个任务，用于验证项目隔离。
4. 电脑 A 授权项目 A；电脑 B 授权项目 A 和可选项目 B。
5. 至少一个任务要求 Codex 或 Claude，至少一个任务要求工具调用或检查结果。

### 必过验收用例

| 用例 | 操作 | 期望结果 |
| --- | --- | --- |
| 双机可见 | 电脑 A、B 同时注册并心跳。 | 总览显示 2 台电脑，电脑详情分别显示不同指纹摘要、Owner、能力和心跳。 |
| 多项目可见 | 中枢存在项目 A、B。 | 有权限 Owner 看到 2 个项目；无项目 B 权限用户只看到项目 A，数量也不包含项目 B。 |
| 项目任务明细 | 项目 A 的两个任务分别由电脑 A、B 执行或领取。 | 项目详情按需求展示两个任务，各自显示 Runner、Agent、进展、模型和 token。 |
| 电脑工作明细 | 打开电脑 A 详情。 | 能看到电脑 A 工作中的 Agent、所属项目、所属需求、任务、工具、Codex / Claude / 模型和最近进展。 |
| 并发租约 | 两台电脑同时尝试领取同一任务。 | 只有一台获得有效租约，另一台看到已占用或领取失败；审计可查。 |
| 上报进展 | Runner 上报 heartbeat、progress_update、model_usage、result_writeback。 | 中枢状态按上报更新；token 计入对应 AgentRun 和任务累计。 |
| 创建项目 | Owner 在工作台创建项目并完成确认或审批。 | 生成 Project 登记记录、初始需求记录、审批或确认记录和 AuditLog；不会直接生成任务租约。 |
| 电脑邀请注册 | Owner 邀请电脑 A，Runner 管理员提交注册。 | 工作台显示邀请范围、申请范围、批准范围；授权前不可调度；全过程有审计。 |
| 工具注册申请 | Tool Owner 登记工具并申请给项目 A 使用。 | 未批准前不可用于任务；批准后显示项目 / Runner / Agent / 数据范围和有效期；全过程有审计。 |
| 项目隔离 | 电脑 A 尝试拉取或上报项目 B 任务。 | 请求失败，不产生 TaskResult，不污染项目 B；中枢显示可读拒绝原因和审计。 |
| 执行监管边界 | 在工作台尝试找派单、转派、暂停执行、修复任务、篡改结果或验收通过入口。 | 工作台可以发起项目/电脑/工具登记，但不存在直接改变执行状态或结果的入口。 |
| 离线恢复 | 电脑 B 执行中断网或停止心跳。 | 中枢在阈值后标记离线或 stale_lease；项目任务显示最后可信进展和恢复建议。 |
| token 缺失 | Runner 上报进展但缺少 token。 | 明细显示 unknown 和缺失来源；验收任务不能被判定为完整通过。 |

### 通过标准

1. 以上必过用例全部通过，并形成可复查验收记录。
2. 任何任务状态都能从项目视角和电脑视角交叉解释一致。
3. 双机证据包含两台不同电脑的注册记录、心跳记录、租约记录、进展记录、AgentRun 和 TaskResult。
4. 项目隔离失败用例必须被拒绝并审计，不能只靠人工观察说“没有看到”。
5. 工作台登记入口和执行监管边界必须由产品验收和权限测试共同确认。
6. 创建项目、电脑注册 / 邀请、工具注册 / 申请必须各自产生确认、审批或拒绝记录，并能从工作台只读复查审计轨迹。
