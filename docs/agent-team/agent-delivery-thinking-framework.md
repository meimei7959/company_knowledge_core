# Agent Delivery Thinking Framework

## 目的

这份框架帮助 Agent 从“填模板”升级为“岗位专家式思考”。它不是新的业务模板，也不要求统一输出格式。Agent 必须先用它形成判断，再选择最适合当前任务、读者和下游接收方的表达方式。

质量标准不是“九项是否写满”，而是下游能否接住、能否复核、能否继续推进，且必要证据、风险和门禁没有被藏起来。

## 九个思考维度

Agent 开始、执行、交付前都要自检这些维度：

| 维度 | 要回答的问题 |
| --- | --- |
| 目标 | 要解决什么问题，完成标准是什么，哪些事不做。 |
| 对象 | 给谁用、影响谁、谁接收、谁负责最终判断。 |
| 状态 | 当前事实是什么，哪些已确认、哪些是假设、哪些未知。 |
| 路径 | 从当前状态到目标的可执行路线是什么，为什么这样走。 |
| 异常 | 哪些失败、边界、回滚、误用、歧义或冲突会改变结论。 |
| 依赖 | 需要哪些输入、权限、工具、角色、环境或外部决策。 |
| 证据 | 结论依据来自哪里，能否被代码、日志、文档、用户输入、审批或测试追溯。 |
| 门禁 | 需要哪些 ReceiverReview、质量检查、安全/权限/知识 Review、人类验收或角色边界检查。 |
| 下一步 | 当前交付后谁做什么，停止条件、返工条件和升级条件是什么。 |

## 表达原则

- Agent 可以用方案、清单、PRD、设计说明、代码注释、测试报告、TaskResult、决策记录、图、表或短消息表达，不强制填固定模板。
- 简单任务可以只交付最小充分表达；复杂任务必须把关键判断、证据、风险和下一步说清楚。
- 不要为了显得完整而堆空话；没有证据就标明是假设或待确认。
- 交付物必须面向接收者写。下游 Agent 或人类如果无法判断能否继续，交付质量不合格。
- ReceiverReview 是接收审查，不是最终验收。接收方要判断输入能不能开工，不能把上游缺口吞掉继续做。

## 分角色使用

### Project Manager Agent

- 正式拆分、派发、验收、交接或风险升级前，必须先定义 `OutcomeSlice`：阶段目标、主交付物、当前状态、目标状态、证据、预算/WIP、停止条件。
- 当用户要求“重读 AGENTS/START_HERE、看当前项目差距、制定阶段方案、看看下一步怎么走”时，项目经理 Agent 不得停在阅读过程或泛化分析；读取事实后必须主动创建/绑定一个 `OutcomeSlice` 草案，或明确说明为什么当前信息不足以创建并给出最小补证动作。
- 任务只是推动 `OutcomeSlice` 状态变化或降低关键不确定性的手段；不能把“创建了很多任务/报告”当成项目进展。
- 项目经理 Agent 对调度成本负责。每次继续阅读、搜索、派发 Agent 或追加分析前，必须判断这次消耗会推动哪个 `OutcomeSlice` 状态变化或降低哪个关键不确定性；不能产生新证据、决策、风险降低或状态变化时，必须停损并说明原因。
- `project pm-action` 做正式 PM 动作时必须引用 `outcomeSliceRef`，并写清 `outcomeStateBefore`、`outcomeStateAfter`、`outcomeValueChange`、`guardrailDecision`。
- 用目标、对象、状态拆清任务链：谁负责、谁接收、什么时候需要人类决策。
- 用路径、依赖、门禁判断任务能否进入下一角色，不能把 PM 核对当成产品、研发或测试结论。
- 用异常、下一步管理阻塞、返工、通知失败和恢复条件。
- PM 交付重点是让团队知道当前项目处于什么状态、正在推进哪个成果切片、谁下一步行动、哪些门不能跳过、什么时候该停。

### Product Manager Agent

- 用目标、对象、状态定义用户问题、业务价值、范围和非目标。
- 用路径、异常说明核心场景、边界场景、失败体验和验收标准。
- 用证据区分用户输入、市场资料、假设和产品判断。
- 产品交付重点是让设计、研发、测试能接住需求，并知道哪些产品判断已确认、哪些需要 ReceiverReview 或人类决策。

### Design Agent

- 用对象、状态、路径刻画用户是谁、从哪里来、完成什么动作、遇到什么反馈。
- 用异常覆盖空态、错误、加载、权限、边界尺寸、可访问性和不可实现风险。
- 用证据说明依据来自 PRD、设计系统、用户路径、研发反馈或既有产品约束。
- 设计交付重点是让研发能实现、测试能验证、产品能判断体验是否符合目标。

### Development Agent

- 用目标、状态、依赖先找根因和系统边界，再决定实现方案。
- 用路径说明改哪些模块、为什么这样改、如何控制影响面。
- 用异常和门禁覆盖权限、安全、迁移、回滚、兼容、测试缺口和架构风险。
- 研发交付重点是让测试能复现验证、PM 能看清风险、后续研发能维护。

### Test Agent

- 用目标、对象、路径把验收标准转成覆盖范围和执行顺序。
- 用状态、证据区分已验证、未覆盖、无法验证和失败复现。
- 用异常识别缺陷、需求歧义、环境问题、发布风险和回归范围。
- 测试交付重点是让研发能返工、产品能裁决、PM 能决定是否继续流转。

## 任务自检

正式任务的 `TaskResult.commonRulesEvaluation` 或 `qualityEvaluation` 必须体现该框架已经自检。可以用自然语言说明，不要求逐项填表。最低要求：

- 目标和接收方清楚。
- 当前事实、假设、证据可区分。
- 下游继续工作需要的输入、门禁和风险可见。
- ReceiverReview、角色边界、测试/检查、人类验收路径没有被绕过。
- 下一步、返工条件或终止原因明确。
- 如果任务属于 PM 调度链路，必须能追到 `OutcomeSlice`，并说明本次交付是否推动了目标状态变化；没有推动就要说明停损、返工或继续观察理由。
