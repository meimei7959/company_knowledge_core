# 源程序材料规范

## 确定性原则

源程序提交版必须能从固定 Git commit 重复生成。脚本先建立完整源文件清单，再按以下顺序形成规范化连续源程序：

1. `source.roots` 的配置顺序；
2. 每个根目录内按 POSIX 相对路径排序；
3. 单个文件内保持原始行顺序；
4. 仅为打印目的去除空白尾部、展开制表符、对超长行做视觉换行；
5. 每个文件边界添加注释形式的文件路径标记，并在报告中说明该标记由排版脚本添加。

禁止先挑选“看起来像核心功能”的 60 页再声称是前后连续页。

## 默认包含

第一方业务源码，例如：

- Java、Kotlin、Swift、Objective-C、Dart；
- Python、Go、Rust、C/C++、C#；
- JavaScript、TypeScript、Vue、React、PHP、Ruby；
- Android 布局 XML、前端模板和与业务逻辑直接相关的 SQL。

## 默认排除

- `.git`、`node_modules`、`vendor`、`Pods`、`.gradle`；
- `dist`、`build`、`target`、`out`、`coverage`；
- 自动生成目录、API 客户端生成代码、ORM 生成代码；
- `.min.js`、source map、压缩包、二进制和媒体文件；
- lock 文件和依赖缓存；
- 测试夹具中的真实数据；
- `.env`、私钥、证书、签名文件、云凭据和生产配置。

## 分页

默认按 50 个打印行分页。若规范化连续源程序：

- 少于 3000 个打印行：全部输出；
- 等于或多于 3000 个打印行：输出前 1500 行和后 1500 行，共 60 页。

超长源代码行可被视觉换行，因此“打印行数”可能大于仓库原始逻辑行数。报告必须同时列出原始非空行数和打印行数。

## 质量检查

- 页眉：软件全称 + 版本号；
- 页码：右上角动态页码；
- A4 纵向、黑白、等宽字体；
- 无截断、乱码、黑块或重叠；
- DOCX 转 PDF 后应逐页视觉检查；
- 代码材料不应包含真实密码、令牌、证书或用户数据。


## AI 与真实性边界

Skill 只能从申请人已经存在的真实第一方源码机械整理，不得生成、补写、改写或拼接拟申报代码。任何 AI 使用事实由申请负责人按当前申请表如实处理。
