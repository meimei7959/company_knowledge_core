# PRD 高质量生成协议

## Purpose

本技能只属于产品经理 Agent。它不创建新的公司级 Agent，也不把 Controller、Reviewer 注册为团队角色。

产品经理 Agent 在写 PRD 时，必须把多视角辩论转化为内部工序：先澄清，再取证，再成案，再反审，再质检，最后交付。

协议分三级，避免所有任务都走重流程：

- `none`：小改动、文案、配置或说明类任务；必须写明不启用原因。
- `light`：默认等级；只要求关键三项摘要：需求澄清器、PRD 质量检查器、交付包生成器。
- `full`：复杂产品、完整上线、跨 Agent 协作、要交开发的 PRD；要求六工序完整证据。

## Triggers

- 用户要求产品方案、PRD、需求文档、需求拆解、测试用例、验收标准或开发交付包。
- 粗想法需要变成完整上线产品需求。
- PRD 需要市场定位、产品定位、商业模式、竞品分析或开发交付。
- 需求存在高不确定性、高协作成本、高质量要求。

## Inputs

- 用户原始需求、补充回答、飞书文档、会议纪要或来源材料。
- 项目目标、业务约束、历史决策、已有 PRD、需求树和验收标准。
- 市场、竞品、用户、政策、标准、技术约束等证据来源。

## Workflow

1. 需求澄清器（light/full 必需）
   - 先用第一性原理拆解：用户是谁、真实问题是什么、价值是什么、成功指标是什么。
   - 再用苏格拉底式提问暴露关键前提：为什么是这个用户、为什么现在做、什么证据能证明成立、什么条件会推翻方案。
   - 明确用户、场景、问题、目标、商业模式、约束、非目标、成功指标。
   - 用苏格拉底式问题追问最高风险缺口。
   - 一次只问 1-3 个阻塞问题；低风险缺口可给默认假设并标注。

2. 证据包生成器（full 必需）
   - 收集或引用市场、竞品、用户、业务、政策、标准、历史决策、技术约束证据。
   - 分开记录事实、推断、假设、待决策事项。
   - 没有来源的市场或竞品判断不得写成事实。

3. 产品方案生成器（full 必需）
   - 按完整上线目标设计产品方案，不默认拆成 MVP、P0、P1。
   - 输出产品定位、目标用户、核心流程、功能范围、边界、依赖、风险和成功指标。
   - 高影响商业、价格、客户承诺、安全、合规事项必须形成决策请求。

4. 反方审查器（full 必需）
   - 产品经理 Agent 自检，而不是注册新的 Reviewer Agent。
   - 挑战问题真实性、用户场景、商业闭环、证据强度、边界异常、可测性、开发可交付性。
   - 审查不通过时，必须回到前序工序修复，不能带病交付。

5. PRD 质量检查器（light/full 必需）
   - 检查是否把任务、模块名、技术实现误写成需求。
   - 检查需求是否包含用户、场景、目标、触发条件、系统响应、边界、验收标准。
   - 检查空状态、加载状态、错误状态、权限、通知、数据、指标、非功能要求是否按场景覆盖。
   - 检查验收标准是否可观察、可测试、可判定。

6. 交付包生成器（light/full 必需）
   - 输出 PRD、需求清单、测试用例、验收清单、开发 Agent 交付包、设计/测试/项目经理协作信息。
   - 每个开发任务必须能追溯到产品需求、验收标准和测试方向。
   - 未决策事项、假设、风险、依赖必须单独列出，不能藏在正文里。

## Outputs

- 需求澄清结论。
- 证据包与假设。
- 产品方案。
- 内部反方审查结果。
- PRD 正文。
- 测试用例。
- 验收清单。
- 开发 Agent 交付包。
- 未决策事项、风险和依赖。

## Quality Gate

- 没有用户和场景，不能定稿 PRD。
- 没有 `requirementClarifier.firstPrinciples` 和 `requirementClarifier.socraticQuestions`，不能通过 PRD 质量门禁。
- 没有产品目标和商业目标，不能输出完整方案。
- 没有证据的市场/竞品结论，必须标注为假设或待调研。
- 没有边界、异常、空状态、加载、错误状态，PRD 不合格。
- 没有可测试验收标准，不能交给开发 Agent。
- 没有测试用例或测试方向，不能算完整交付包。
- 技术方案、数据库字段、接口路径、实现步骤不能冒充产品需求。
- 公司级 Agent 列表不得因为本技能增加 Controller、Reviewer 或其他临时角色。
- 默认使用 `light`，复杂完整上线任务使用 `full`，小改动使用 `none` 时必须记录原因。

## Failure Routes

- 需求不清：回到 `requirement-clarification`，只问 1-3 个阻塞问题。
- 缺少证据：创建调研任务或把结论标注为假设。
- 验收不可测：回到 PRD 范围定义并重写验收标准。
- 缺少测试用例或开发交付包：不能标记完整交付。
- 试图新增 Controller/Reviewer 公司级 Agent：阻止注册，改为产品经理 Agent 内部质量步骤。

## 中文输出要求

所有输出默认使用中文，除行业固定英文缩写、代码标识、接口名、文件路径外，不使用英文标题或英文说明。

标准交付结构：

1. 需求澄清结论
2. 证据包与假设
3. 产品方案
4. 反方审查结果
5. PRD 正文
6. 测试用例
7. 验收清单
8. 开发 Agent 交付包
9. 未决策事项、风险和依赖

`prdQualityProtocol.value` 建议格式：

```json
{
  "requiredProtocolLevel": "light",
  "requirementClarifier": {
    "firstPrinciples": {
      "user": "目标用户",
      "problem": "真实问题",
      "value": "产品价值",
      "successMetric": "可观察成功指标"
    },
    "socraticQuestions": [
      "为什么这个用户最需要解决？",
      "什么证据证明这个问题值得做？"
    ],
    "summary": "已明确用户、场景、问题和成功指标。"
  },
  "prdQualityChecker": "已确认需求可测试，且未混入技术实现。",
  "deliveryPackGenerator": "已给出验收方向和开发交付要点。"
}
```

## 协作边界

- 项目经理 Agent 仍负责项目计划、任务编排、进度、风险和关闭。
- 设计 Agent 仍负责信息架构、交互、界面规范和可用性细化。
- 开发 Agent 仍负责技术方案和实现。
- 测试 Agent 仍负责测试计划、自动化、缺陷和质量结论。
- 产品经理 Agent 只输出产品侧完整需求、验收方向和跨 Agent 交付包。
