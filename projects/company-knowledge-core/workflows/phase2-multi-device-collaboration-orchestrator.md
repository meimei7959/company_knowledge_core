# 阶段二多设备协作任务编排器

## 定位

这是项目经理 Agent 的 Workflow，不是共享 Skill。它负责规定阶段二多设备协作的岗位顺序、任务门禁、交接证据、失败回路和最终验收条件。

各岗位仍使用自己的专业 Skill：

- 产品经理 Agent：需求澄清、PRD/范围定义、产品信息架构。
- 设计 Agent：`web-design-engineer`、`ui-ux-pro-max`、UI/UX 评审、交互状态说明。
- 架构师 Agent：架构与技术方案设计、代码架构评审。
- 研发 Agent：根因分析、实施计划、编码实现。
- 测试 Agent：测试计划、回归、端到端验收。

## 编排顺序

1. 产品经理 Agent 输出阶段二产品需求包。
2. 产品经理 Agent 输出产品信息架构。
3. 设计 Agent 基于产品信息架构输出 UI/交互设计。
4. 架构师 Agent 基于产品需求、产品信息架构、UI/交互设计输出或修订技术方案。
5. 产品经理 Agent 复核技术方案是否满足需求、信息架构和体验目标。
6. 研发 Agent 在产品复核通过后实现。
7. 测试 Agent 验证多设备 Runner、权限/配对、路由、TaskResult 写回和用户可读工作台。
8. 项目经理 Agent 汇总证据，完成 PM 流程验收或记录阻塞。

## 角色边界

- 项目经理 Agent 不替产品、设计、架构、研发、测试给结论。
- 产品经理 Agent owns 信息架构；设计 Agent consumes 信息架构。
- 设计 Agent owns UI 设计、交互设计、视觉层级、状态表达和体验质量门禁。
- 架构师 Agent 不直接实现代码。
- 研发 Agent 不跳过产品复核和设计门禁。
- 测试失败必须回研发 Agent 修复，不能由 PM 或主线程代修。

## 研发准入

研发任务只能在以下证据全部存在后启动：

- 产品需求包。
- 产品信息架构。
- UI/交互设计。
- 架构技术方案。
- 产品经理对技术方案的通过结论。
- 项目经理确认任务卡、验收标准和失败回路齐全。

## 用户体验门禁

工作台是给人看的操作台，不是工程调试面板。

- 主界面必须使用中文业务含义。
- 用户必须能看懂当前项目、同事接入状态、任务路由去向、卡住原因、下一步动作。
- 内部 id、raw status、capability code、文件路径、sessionId、runnerId、deviceId 不得作为主信息展示。
- 技术值只能放在证据、审计、技术详情或折叠区。

## 多设备验收门禁

- 至少能登记两台设备或一台真实设备加一台明确标注的模拟设备。
- 每个 Runner 有项目范围、岗位范围、能力范围、工具范围、仓库范围和数据范围。
- 调度决策能解释为什么派给某个 Runner。
- 租约、心跳、重试、转派、取消、结果写回都有证据。
- 真实双 host 验收不能被本机双 Runner 模拟替代，除非产品经理明确接受为阶段性替代证据。

## 失败回路

- 产品范围不清：退回产品经理 Agent。
- 信息架构缺失：退回产品经理 Agent。
- UI/交互设计缺失或不可用：退回设计 Agent。
- 技术方案不满足产品/设计：退回架构师 Agent。
- 实现缺陷：退回研发 Agent。
- 测试失败：创建缺陷任务并交研发 Agent。
- 权限、共享中枢或真实同事设备缺失：项目经理 Agent 记录阻塞并请求人类/管理员动作。

## 关闭标准

项目经理 Agent 只能在以下条件满足后关闭阶段二：

- 所有岗位 TaskResult 均存在。
- 测试 Agent 通过多设备路由和用户可读工作台验证。
- 产品经理 Agent 通过最终产品验收。
- PM 复核没有未登记阻塞、未关闭缺陷或未说明的范围差异。
