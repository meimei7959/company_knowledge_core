# 阶段二方案二中枢监管编排器

## 目标

把已部署的 Knowledge API 地址作为项目中枢，让多台电脑以 Runner 身份接入同一个中枢，并让工作台承担受控登记入口和真实运行监管。

工作台可以发起项目创建、电脑注册、工具注册或工具注册申请。运行中的任务调度、修复、结果写回和验收结论仍由中枢、Runner、Agent 工作流和审批链完成，工作台只展示监管状态，不直接篡改执行结果。

## 必须可见

- 当前中枢下有几个项目。
- 当前中枢下有几台电脑接入。
- 每个项目的任务明细：对应需求、当前进展、负责 Agent、执行电脑、使用模型、消耗 token、使用工具、最后更新时间、验收状态。
- 每台电脑的明细：在线状态、当前项目、当前任务、有哪些 Agent 在工作、使用 Codex 还是 Claude、使用了哪些工具、模型和 token 消耗、最近心跳、异常原因。
- 项目创建、电脑注册、工具注册的申请状态、审批状态和审计记录。
- Runner 注册和进展上报是否成功。

## 编排顺序

1. 产品经理 Agent 输出阶段二方案二补充 PRD，锁定工作台登记入口、Runner 注册、进展上报、权限和验收口径。
2. 架构 Agent 输出中枢 API、Runner 协议、幂等、离线恢复、项目隔离和审计方案。
3. 设计 Agent 基于产品信息架构输出中文、Codex 风格、用户可理解的工作台设计。
4. 产品经理 Agent 复核架构和设计是否满足用户可理解、受控登记、只读监管和真实双机闭环。
5. 研发 Agent 实现中枢 API、Runner 客户端上报、工作台登记入口、监管 read model、工作台展示和接入文档。
6. 测试 Agent 验证 API、CLI、工作台、权限隔离、异常恢复和真实双机证据。
7. 项目经理 Agent 汇总流程证据，产品经理 Agent 做产品验收，项目经理 Agent 做过程验收。

## 研发准入

研发不得在以下材料缺失时开工：

- 产品补充 PRD。
- 架构技术方案。
- 设计方案。
- 产品经理对架构和设计的复核结论。

## 项目经理边界

- 项目经理 Agent 负责调度、对账、风险、阻塞、证据和验收组织。
- 项目经理 Agent 不得把研发实现、测试执行、产品验收结论直接替岗位 Agent 完成。
- 如果项目经理 Agent 在协调中产生实现补丁，该补丁只能视为草稿，必须交给研发 Agent 接管、审查、保留或重写。
- 研发 Agent 未接管并输出 TaskResult 前，该实现不能算完成。
- 测试 Agent 未验证前，该实现不能算通过。
- 影响用户体验的变更，产品 Agent 未验收前不能算产品完成。

## 工作台边界

- 工作台的登记入口可操作，包括创建项目、生成电脑接入邀请、提交工具注册或工具注册申请。
- 登记入口必须经过权限校验、用户确认、幂等键和 AuditLog。
- 工作台不得保存原始 token。
- 工作台不得直接修改任务状态。
- 工作台不得直接派单、转交任务、禁用 Runner、修复任务、篡改 TaskResult 或批准验收结论。
- 工作台不得把内部 id、文件路径、sessionId、runnerId、deviceId 作为主信息展示；必须优先展示用户可理解名称。
- 工作台可以提供复制接入命令、查看证据链接、查看异常说明；会改变登记状态的动作必须进入授权流程。

## Runner 注册和上报边界

每台电脑必须通过中枢授权后注册为 Runner。

Runner 必须上报：

- 设备身份、设备名称、系统类型、工作目录摘要。
- 可服务项目、可用 Agent、可用工具、能力标签。
- 当前会话使用 Codex 或 Claude。
- 当前任务、所属需求、任务状态、进度摘要。
- 使用模型、输入 token、输出 token、总 token、工具调用摘要。
- 心跳时间、负载、错误和恢复状态。

Runner 不得上报：

- 原始密钥、Token、Cookie、SSH private key。
- 未脱敏的客户隐私内容。
- 本地完整敏感路径作为用户主信息。

## 关闭标准

- 两台真实电脑接入同一个部署中枢。
- 中枢显示至少两个 Runner 在线或可解释离线。
- 一个真实任务从项目经理 Agent 派发到其中一台 Runner。
- Runner 上报进展、模型、token、工具和 Agent 状态。
- Runner 写回 TaskResult。
- 测试 Agent 基于中枢证据验收通过。
- 产品经理 Agent 验收工作台内容用户可理解、登记入口可用、运行监管不越权。
- 项目经理 Agent 验收调度、审计、异常恢复和证据闭环。
