# 阶段二多设备 Runner 协作工作台设计规范

## UI 与交互返工补充

本设计规范的可实现 UI/交互细化已补充到 `projects/company-knowledge-core/design/phase2-multi-device-runner-workbench-ui-interaction-revision.md`。该返工稿按 `projects/company-knowledge-core/workflows/phase2-multi-device-collaboration-orchestrator-orchestrator.md` 和已修正的 `docs/agent-team/design-agent-role-and-skill-pack.md` 执行，重点覆盖页面布局、区域结构、视觉层级、组件清单、按钮/弹窗/抽屉、邀请同事流程、配对授权流程、设备/执行器卡片状态、任务路由状态、异常/权限/空/加载/只读降级状态、中文文案示例、窄屏布局、研发标注和测试验收项。

## 设计结论

阶段二工作台面向真实用户，不是工程调试面板。最高优先级是用户看得懂、知道当前项目由谁在做、哪台同事电脑可用、任务为什么被路由到某个执行器、哪里需要授权或人工处理。

页面主信息必须使用中文业务含义表达。`internal id`、`runtimeMetrics`、`sessionId`、`runnerId`、`deviceId`、`capability code`、`raw status`、本地文件路径、接口错误码、堆栈、租约原始字段只能放入“技术详情/证据”折叠区，不能作为标题、按钮、状态、列表主列、空状态或错误提示的主信息。

## 适用范围

- 阶段二“同事电脑接入同一项目中枢 / 多设备 Runner 协作闭环”。
- 适用于桌面工作台、Web 控制台、窄屏查看模式的同一信息架构。
- 负责页面设计、中文命名、状态表达、交互流、降级规则和研发验收清单。
- 不定义 Agent Ring 内部实现，不暴露调试字段，不替代 API 协议文档。

## 用户目标

| 用户 | 主要问题 | 页面必须回答 |
| --- | --- | --- |
| 项目负责人 | 同事电脑有没有接入，项目能不能继续跑。 | 当前项目可用执行器、正在处理的任务、需要我处理的异常。 |
| 协作同事 | 我的电脑是否已加入项目，能执行什么。 | 配对状态、授权范围、当前是否被分配任务、如何退出或暂停。 |
| 项目经理 / 调度负责人 | 哪些任务正在排队、执行、等待授权、失败。 | 路由状态、阻塞原因、下一步动作、证据入口。 |
| 审批人 / 管理员 | 是否可以允许这台电脑接入项目。 | 申请人、电脑名称、项目范围、可执行任务类型、风险提示。 |

## 导航位置

阶段二入口放在现有主导航 `Runner` 下，中文名升级为“协作设备”。保留 Runner 作为技术对象名，只在技术详情或接口文档里出现。

| 导航层级 | 中文显示 | 说明 |
| --- | --- | --- |
| 主导航 | 协作设备 | 查看同一项目下同事电脑、执行器、授权和任务路由。 |
| 项目页快捷入口 | 接入同事电脑 | 当前项目无可用同事设备或需要扩容时显示。 |
| 任务详情侧栏 | 可执行设备 | 显示该任务可路由到哪些设备，以及为什么未路由。 |
| 审批/权限页 | 设备接入申请 | 审批新电脑加入项目、扩展能力或恢复授权。 |
| 异常恢复页 | 设备与任务异常 | 处理离线、授权失效、任务卡住、只读降级。 |

导航不新增“调试”“运行指标”“租约”这类工程词。用户路径以项目为中心：

```txt
选择项目 -> 协作设备 -> 邀请同事 / 配对授权 -> 设备可用 -> 任务路由 -> 结果与证据
```

## 项目选择器关系

项目选择器是页面的第一上下文。所有设备、执行器、任务路由、授权申请都必须绑定当前项目显示。

| 位置 | 行为 |
| --- | --- |
| 顶部栏左侧 | 显示当前项目名称、项目状态、可切换项目。 |
| 切换项目后 | 设备列表、邀请链接、任务路由、权限提示全部刷新到新项目。 |
| 无项目权限 | 页面进入只读降级，显示“你暂无此项目的协作设备权限”。 |
| 多项目设备 | 同一电脑可接入多个项目，但主列表只显示当前项目授权范围。 |
| 全局视角 | 管理员可在筛选器切换“全部项目”，但默认仍按当前项目展示。 |

项目选择器不可显示项目内部路径、配置文件路径、仓库本地路径作为主信息。需要说明来源时，显示“来自项目资料库”“来自代码仓库授权”，具体路径放入技术详情。

## 页面信息架构

### 页面骨架

1. 顶部上下文栏：项目选择器、项目协作状态、主要动作“邀请同事”。
2. 状态总览：可用设备、执行中任务、等待授权、异常设备。
3. 设备/执行器列表：按业务可读字段展示同事电脑和执行器。
4. 任务路由看板：排队、已分配、执行中、等待授权、需人工处理、已完成。
5. 授权与配对记录：待确认、已授权、已撤销、已过期。
6. 证据与技术详情：默认折叠，面向研发和审计。

### 信息优先级

| 优先级 | 显示内容 | 示例 |
| --- | --- | --- |
| P0 | 用户必须立刻理解的业务状态 | “张三的 Mac 可以接任务”“任务等待李四授权” |
| P1 | 下一步动作 | “邀请同事”“重新发送授权”“转交给其他设备” |
| P2 | 解释原因 | “这台电脑缺少代码仓库权限，不能执行开发任务。” |
| P3 | 证据入口 | “查看执行证据”“查看授权记录” |
| P4 | 技术详情 | `runnerId`、`sessionId`、原始状态、接口响应、路径 |

## 邀请同事入口

### 入口位置

| 场景 | 入口 | 显示规则 |
| --- | --- | --- |
| 有项目权限 | 顶部主按钮“邀请同事” | 始终可见。 |
| 无可用设备 | 空状态主按钮“邀请同事接入电脑” | 替代列表空白。 |
| 任务缺执行器 | 任务卡片按钮“邀请可执行同事” | 带任务类型上下文。 |
| 审批页 | 次按钮“复制邀请说明” | 供管理员转发。 |

### 邀请弹窗

标题：邀请同事电脑加入项目

必填信息：

- 项目：显示当前项目中文名称。
- 允许执行的工作：设计、开发、测试、知识整理、项目管理等业务名称。
- 有效期：默认 24 小时，可选 1 小时、24 小时、7 天。
- 审批人：需要审批时显示中文姓名或角色。

主按钮：

- “生成邀请”
- “复制邀请链接”
- “发送到飞书”

提示文案：

- “对方接受后，需要完成电脑配对和授权确认，才会开始接任务。”
- “邀请不会授予超出当前项目的资料、仓库或工具权限。”

禁止主显示：

- 邀请 token。
- `pairingProofRef`。
- API endpoint。
- 本地启动命令。
- 原始权限 scope。

这些内容如研发需要，放入“技术详情”折叠区。

## 设备/执行器列表

### 列表主列

| 列 | 用户可读名称 | 显示规则 |
| --- | --- | --- |
| 设备 | 同事电脑 | “张三的 MacBook Pro”“设计组 Windows 台式机”。 |
| 当前状态 | 可用状态 | 可接任务、正在执行、等待授权、离线、需处理。 |
| 可做工作 | 工作类型 | 设计、开发、测试、资料整理、审批辅助。 |
| 当前任务 | 正在处理 | 显示任务标题或“暂无任务”。 |
| 授权范围 | 项目范围 | 当前项目、指定任务、只读资料、不可访问代码。 |
| 最近在线 | 活跃时间 | “刚刚”“10 分钟前”“昨天 18:30”。 |
| 操作 | 用户动作 | 查看、暂停接单、重新授权、转交任务。 |

### 列表密度

- 默认密度：每行 56-64px，高频运维可扫描。
- 关键状态使用标签，不用大面积彩色卡片。
- 单行最多 2 个主标签：可用状态 + 授权风险。
- 操作收敛为主按钮 + 更多菜单，避免每行按钮过多。
- 设备详情用右侧抽屉，不跳离当前项目上下文。

### 设备详情抽屉

分区：

1. 设备概况：同事、电脑名称、可做工作、当前状态。
2. 当前项目授权：资料、仓库、工具、任务类型。
3. 当前任务：任务标题、进度、预计下一步。
4. 近 10 次执行记录：业务结果、证据、问题。
5. 技术详情/证据：默认折叠，含内部字段、原始状态、接口响应引用。

## 配对授权流程

### 主流程

```txt
邀请同事 -> 同事打开邀请 -> 选择本机执行器 -> 发起配对 -> 项目侧确认 -> 授权生效 -> 设备可接任务
```

### 用户界面步骤

| 步骤 | 页面标题 | 主状态 | 主动作 |
| --- | --- | --- | --- |
| 1 | 邀请同事电脑加入项目 | 邀请未发送 | 生成邀请 |
| 2 | 等待同事接受邀请 | 邀请已发送 | 复制邀请 / 撤销邀请 |
| 3 | 同事电脑请求接入 | 等待确认 | 同意接入 / 拒绝 |
| 4 | 确认授权范围 | 需要选择范围 | 确认授权 |
| 5 | 设备已加入项目 | 可接任务 | 查看设备 |

### 授权确认内容

确认页必须用中文业务词说明风险：

- 这台电脑将加入哪个项目。
- 由谁发起接入。
- 电脑显示名称是什么。
- 可以执行哪些类型的任务。
- 可以读取哪些项目资料。
- 是否可以访问代码仓库。
- 是否可以调用已批准工具。
- 授权有效期和撤销入口。

禁止只显示：

- `scope: repo.read`
- `capability: dev.codex.local`
- `runnerId`
- `deviceId`
- `leasePolicy`

技术字段可折叠显示，标题固定为“技术详情/证据”。

## 任务路由状态

任务路由状态必须解释“为什么在这里”和“下一步谁做”。不要把 raw status 直接暴露给用户。

| 用户状态 | 业务含义 | 主提示 | 可用动作 |
| --- | --- | --- | --- |
| 待分配 | 任务已进入队列，正在找合适设备。 | 正在匹配可执行同事电脑。 | 查看条件、手动指定。 |
| 可执行 | 已找到设备，等待调度。 | 可由张三的 Mac 执行。 | 指派、稍后执行。 |
| 等待授权 | 找到设备，但权限不足。 | 需要李四确认代码仓库访问。 | 发起授权、换设备。 |
| 执行中 | 设备正在处理任务。 | 张三的 Mac 正在执行开发任务。 | 查看进度、查看证据。 |
| 等待回传 | 执行结束，结果正在写回。 | 正在同步结果和证据。 | 刷新、查看已回传部分。 |
| 需人工处理 | 自动流程无法继续。 | 任务需要人工确认后继续。 | 查看原因、转交、取消。 |
| 已完成 | 结果已写回并可验收。 | 已生成结果和证据。 | 去验收、查看证据。 |
| 已暂停 | 人工或策略暂停。 | 任务已暂停，不会继续路由。 | 恢复、转交。 |
| 已失败 | 执行未完成。 | 任务执行失败，已保留证据。 | 重试、转交、提交问题。 |

### 路由解释

每条路由记录显示三行以内：

- 分配给谁：同事电脑中文名。
- 为什么分配：具备对应工作类型、项目授权、空闲。
- 当前卡点：无卡点 / 等待授权 / 设备离线 / 结果待回传。

技术字段只在证据折叠区显示：

- `claimId`
- `leaseId`
- `runnerId`
- `routingDecisionId`
- `raw status`
- `runtimeMetrics`

## 异常与权限提示

### 提示原则

- 先说业务影响，再说可操作下一步。
- 不用“403”“permission denied”“stale lease”做主标题。
- 不把责任推给用户无法理解的系统对象。
- 每个错误必须有一个主动作或明确说明“只能联系管理员”。

### 标准提示

| 场景 | 标题 | 正文 | 主动作 |
| --- | --- | --- | --- |
| 无项目权限 | 你暂无此项目的协作设备权限 | 可以查看项目公开信息，但不能邀请同事或管理设备。 | 申请权限 |
| 邀请过期 | 邀请已过期 | 为了保护项目资料，邀请链接已失效。请重新生成邀请。 | 重新生成邀请 |
| 配对待审批 | 等待项目负责人确认 | 同事电脑已提交接入申请，确认后才能接任务。 | 查看申请 |
| 设备离线 | 同事电脑暂时离线 | 当前任务不会继续在这台电脑上执行，可等待恢复或转交。 | 转交任务 |
| 权限不足 | 这台电脑缺少必要授权 | 任务需要访问代码仓库或工具，请补充授权或换设备。 | 发起授权 |
| 授权失效 | 设备授权已失效 | 授权到期或被撤销，设备已停止接新任务。 | 重新授权 |
| 写回失败 | 结果暂未同步完成 | 执行可能已完成，但结果和证据还没写回项目中枢。 | 重试同步 |
| 只读降级 | 当前为只读模式 | 系统无法确认写入权限，只展示最近可用状态。 | 重新连接 |

## 空状态、加载、失败、只读降级

| 状态 | 页面表现 | 文案 | 动作 |
| --- | --- | --- | --- |
| 首次空状态 | 居中说明 + 主按钮 | 还没有同事电脑加入这个项目 | 邀请同事 |
| 无可执行设备 | 状态条 + 任务提示 | 当前没有能处理此类任务的设备 | 邀请可执行同事 |
| 加载中 | 骨架屏 + 保留页面结构 | 正在读取项目协作设备 | 无，超过 8 秒显示重试 |
| 局部加载 | 行内骨架 | 正在更新状态 | 不遮挡已知信息 |
| 加载失败 | 错误条 + 保留旧数据 | 暂时无法更新设备状态 | 重试 |
| 写入失败 | 顶部提示 + 失败动作回滚 | 操作没有保存，请重试 | 重试 / 查看原因 |
| 只读降级 | 顶部黄色提示 + 禁用写按钮 | 当前为只读模式，只展示最近一次同步结果 | 重新连接 / 申请权限 |
| 数据过期 | 时间提示 + 刷新入口 | 状态可能不是最新，最近更新于 10 分钟前 | 刷新 |

只读降级时，按钮禁用文案必须说明原因，例如“无权限邀请同事”“只读模式下不能暂停设备”。不能只显示 disabled。

## 中文命名规范

| 技术词 | 用户界面中文 | 使用规则 |
| --- | --- | --- |
| Runner | 协作设备 / 执行器 | 导航用“协作设备”，详情可称“执行器”。 |
| Agent Ring | 执行网络 | 只在说明外部执行能力时使用。 |
| Lease | 任务占用 | 主界面不用“租约”。 |
| Claim | 领取任务 | 用户侧说“任务已分配”。 |
| Heartbeat | 在线状态 | 不说“心跳”。 |
| Capability | 可做工作 | 显示设计、开发、测试等业务名。 |
| Scope | 授权范围 | 显示项目资料、代码仓库、工具。 |
| Session | 执行会话 | 主界面不用，证据区可显示。 |
| Runtime metrics | 运行指标 | 主界面不用，技术详情可显示。 |
| Raw status | 原始状态 | 只能在技术详情显示。 |
| AuditLog | 操作记录 | 用户界面显示“操作记录”。 |
| TaskResult | 任务结果 | 用户界面显示“结果与证据”。 |

中文句式：

- 标题用名词短语：“协作设备”“设备接入申请”“任务路由”。
- 状态用可理解短句：“等待授权”“正在执行”“结果待同步”。
- 按钮用动词短语：“邀请同事”“确认授权”“转交任务”“查看证据”。
- 错误先给业务影响：“任务暂时不能继续”，再给原因。

## 视觉密度

- 页面是工作台，不是营销页。信息密度偏高，但留出清晰分组。
- 顶部总览使用 4 个紧凑指标，不使用大英雄区。
- 设备列表优先表格/列表，卡片只用于空状态、授权申请、移动端设备块。
- 颜色克制：绿色=可用，蓝色=执行中，黄色=等待/需授权，红色=失败/阻塞，灰色=离线/禁用。
- 每屏首要目标：用户不用滚动就能看到项目、设备总数、异常数、邀请入口。

## 按钮、标签、卡片规范

### 按钮

| 类型 | 用途 | 示例 |
| --- | --- | --- |
| 主按钮 | 页面最重要动作 | 邀请同事、确认授权、转交任务 |
| 次按钮 | 辅助动作 | 复制邀请、查看记录、刷新状态 |
| 危险按钮 | 撤销或停止 | 撤销授权、暂停接单 |
| 图标按钮 | 高频轻动作 | 刷新、更多、展开详情 |

按钮文字必须中文。禁止“Approve Runner”“Retry lease”“Copy token”。

### 标签

| 类型 | 示例 | 规则 |
| --- | --- | --- |
| 状态标签 | 可接任务、执行中、等待授权、离线 | 每个对象最多一个主状态。 |
| 风险标签 | 权限不足、授权将过期、只读 | 与主状态并列，不替代主状态。 |
| 工作标签 | 设计、开发、测试、资料整理 | 来自能力码映射，不显示能力码。 |

### 卡片

卡片只用于：

- 空状态。
- 授权申请。
- 移动端设备块。
- 任务路由摘要。

卡片内不再嵌套卡片。卡片标题必须是业务对象名，不得是内部 ID。

## 移动端与窄屏约束

| 宽度 | 布局 |
| --- | --- |
| >= 1200px | 左主列表 + 右任务路由/详情抽屉。 |
| 768-1199px | 单列列表，任务路由放在列表下方，详情用抽屉。 |
| < 768px | 设备卡片流，隐藏非关键列，操作进更多菜单。 |

窄屏必须保留：

- 项目选择器。
- 当前协作状态。
- 邀请同事入口。
- 设备状态。
- 任务路由状态。
- 权限/异常提示。

窄屏可以折叠：

- 历史记录。
- 技术详情。
- 授权细项。
- 运行证据列表。

移动端按钮不得挤压成省略号主动作。主动作必须完整显示中文，例如“邀请同事”“确认授权”。

## 不可暴露的内部字段

下列字段不得作为主信息显示：

| 内部字段 | 禁止位置 | 可出现位置 |
| --- | --- | --- |
| `internal id` | 标题、列表主列、按钮、状态 | 技术详情/证据 |
| `runnerId` | 设备名称、授权申请标题 | 技术详情/证据 |
| `deviceId` | 设备名称、配对页标题 | 技术详情/证据 |
| `sessionId` | 任务标题、进度说明 | 技术详情/证据 |
| `runtimeMetrics` | 状态总览、异常提示 | 技术详情/证据 |
| `capability code` | 可做工作标签 | 技术详情/证据 |
| `raw status` | 状态标签、错误标题 | 技术详情/证据 |
| 本地文件路径 | 项目来源、证据入口主文案 | 技术详情/证据 |
| API 错误码 | 错误标题 | 技术详情/证据 |
| token / secret | 任何页面 | 不显示；只显示“凭据已保存/已失效” |

如果没有可读名称，界面必须生成临时中文名，例如“未命名的同事电脑”，并提示用户补充名称。不能退回显示 `runner-xxx`。

## 证据入口设计

证据入口必须让用户理解“看它能确认什么”。

| 入口名称 | 业务含义 |
| --- | --- |
| 查看执行证据 | 看任务做了什么、输出在哪里、是否可验收。 |
| 查看授权记录 | 看谁允许了这台电脑接入项目。 |
| 查看操作记录 | 看谁暂停、转交、重新授权。 |
| 查看技术详情 | 给研发排查使用，默认折叠。 |

证据摘要显示：

- 结果状态。
- 产出名称。
- 操作人或执行设备中文名。
- 时间。
- 可读原因。

技术详情显示：

- 原始对象引用。
- 内部 ID。
- 原始状态。
- 已脱敏错误。
- 审计引用。

## 可执行页面验收清单

### 架构师验收

- [ ] 页面入口在“协作设备”，并与现有 `Runner` 技术对象关系清楚。
- [ ] 项目选择器是所有设备、任务路由、授权申请的唯一主上下文。
- [ ] 邀请同事、配对授权、任务路由、结果证据形成闭环。
- [ ] 设备接入不绕过 Scheduler、Agent Ring、TaskResult、AgentRun、AuditLog 边界。
- [ ] 授权范围按项目、任务类型、资料、仓库、工具拆分，可审计、可撤销。
- [ ] 只读降级不允许写操作，但保留最近可读状态。
- [ ] 技术详情折叠区能满足研发排障，不污染主界面。
- [ ] 移动/窄屏保留关键任务流，不依赖宽屏表格才能完成授权。

### 研发验收

- [ ] 设备列表主列无 `runnerId`、`deviceId`、`sessionId`、raw status。
- [ ] 所有能力码映射为中文工作类型：设计、开发、测试、资料整理等。
- [ ] 所有状态映射为中文业务状态：可接任务、等待授权、执行中、离线、失败等。
- [ ] 邀请弹窗不展示 token、API endpoint、本地命令作为主信息。
- [ ] 授权确认页显示申请人、电脑名称、项目、工作类型、资料/仓库/工具范围。
- [ ] 路由状态显示“为什么分配”和“下一步谁处理”。
- [ ] 错误提示先显示业务影响，再显示用户可做动作。
- [ ] 空状态、加载、失败、只读降级都有中文文案和明确动作。
- [ ] 禁用按钮有中文原因，不只依赖灰色样式。
- [ ] 技术详情默认折叠，且敏感值脱敏。
- [ ] 移动端主动作完整显示，不被截断。
- [ ] 每个写操作能产生操作记录或审计引用。

### 用户可读性逐项检查

- [ ] 页面标题用户能一眼知道这是“同事电脑协作”，不是调试台。
- [ ] 每个按钮都是中文动词短语，用户知道点后会发生什么。
- [ ] 每个状态都是中文业务状态，用户不用理解 Agent Ring 内部机制。
- [ ] 每个错误提示都说明业务影响、原因和下一步。
- [ ] 每个证据入口说明“看这个能确认什么”。
- [ ] 每个空状态都说明为什么为空，以及用户可以做什么。
- [ ] 列表第一列显示同事电脑可读名称，不显示内部 ID。
- [ ] 可做工作显示业务类别，不显示 capability code。
- [ ] 授权范围显示资料、仓库、工具等业务对象，不显示 scope 字符串。
- [ ] 路由说明显示“分配给谁”和“为什么”，不显示 claim/lease 原始字段。
- [ ] 技术详情折叠后，普通用户仍能完成邀请、授权、转交、验收。
- [ ] 无权限用户能理解自己处于只读模式，以及如何申请权限。
- [ ] 窄屏上不丢失项目选择器、邀请入口、设备状态和任务路由。
- [ ] 无可读名称时显示“未命名的同事电脑”，并引导补充名称。
- [ ] 任何主界面都不出现本地文件路径、token、secret、接口堆栈。

## 交付结论

阶段二页面应把“多设备 Runner 协作”翻译成用户能操作的“同事电脑协作”。设计落地时，主界面只表达项目、同事电脑、可做工作、授权、任务、证据和下一步动作；技术对象全部退到可折叠证据层。这样既保留 Agent Ring 可审计闭环，也让真实用户能独立完成接入和协作。
